网站系统挂掉故障修复报告
发布于：2026年03月22日 更新于：2026年03月18日 15:22

️ 故障排查与修复报告 (Incident Report)

  1. 故障现象

    • 容器状态：MySQL 容器（ID: 0f0a8da7e2a8）处于 Restarting (137) 无限重启循环中。
    • 重启频率：每隔几秒尝试启动一次，但始终无法进入 Healthy 状态。
    • 关联影响：依赖该数据库的 Halo 博客系统等应用无法正常连接，报错并持续尝试重连。

  2. 核心问题分析 (Root Causes) 经过深入排查，发现该故障由以下三个因素叠加导致：

    A. 物理内存耗尽 (OOM - Out of Memory)

    • 发现：服务器物理内存为 3.6G，但在排查时可用内存（Available）仅剩约 150MB，且系统未配置 Swap 交换分区。
    • 影响：MySQL 8.2 在启动时需要初始化 InnoDB 缓冲池等内存结构。由于内存极度紧张，进程在初始化过程中被系统内核（OOM Killer）直接杀死（退出码 137 通常表示被 SIGKILL 强制停止）。

    B. Docker 容器状态“半死不活” (Zombie Container State)

    • 发现：即使执行 docker stop，后台仍残留一个 PID 为 15107 的 mysqld 进程和对应的 containerd-shim。
    • 影响：Docker 引擎由于之前的异常崩溃，失去了对该子进程的有效控制。当 Docker 尝试重启容器时，新进程因为旧进程占用了 /var/lib/mysql 下的数据文件锁（File Lock）而启动失败，陷入死循环。

    C. 启动自增压力 (Binlog Explosion)

    • 发现：数据目录下堆积了多达 28,564 个 tiny binlog 文件。
    • 原因：MySQL 每次尝试启动都会创建一个新的 binlog。由于之前的无限重启循环，产生了数万个碎小文件，这导致 MySQL 每次启动时的初始化扫描变得异常缓慢。

  1. 修复方案与操作记录

    第一步：数据安全保护

    • 操作：在进行任何高风险操作前，对 /opt/1panel/apps/mysql/mysql/data 进行了压缩备份（tar.gz），确保数据绝对安全。

    第二步：清理“幽灵”进程

    • 操作：手动强制杀死了残留的 mysqld 和 containerd-shim 进程，释放了数据文件锁，为干净启动扫清障碍。

    第三步：内存扩容 (添加 Swap)

    • 操作：创建并启用了 1GB 的 Swap 交换分区，并写入 /etc/fstab 确保重启后依然有效。
    • 效果：为系统提供了必要的缓冲空间，避免 MySQL 再次因为瞬时内存峰值被系统强制杀死。

    第四步：优化 MySQL 配置

    • 操作：在 my.cnf 中明确设置了 innodbbufferpool_size=256M。
    • 效果：限制了 InnoDB 的内存胃口，使其在低内存环境下能更稳定地运行。

  1. 当前状态验证

    • MySQL 状态：容器已恢复 Up 状态，显示 ready for connections。
    • 应用状态：Halo 容器已成功连接数据库并完成数据同步（日志显示 Started request synchronizer）。
    • 系统稳定性：Swap 分区已开始承载部分非活跃内存，系统整体运行平稳。

  2. 运维建议

    1. 定期清理 Binlog：目前的 binlog 数量过多，虽然不影响当前运行，但建议进入数据库执行 PURGE BINARY LOGS 清理过旧日志。
    2. 监控内存：该服务器运行了多个 Java 应用（Halo）和邮局系统（Coremail），内存压力较大，建议保持 Swap 开启。
    3. 1Panel 配置检查：建议在 1Panel 面板中检查 MySQL 的内存配额设置，确保其与 my.cnf 中的限制保持一致。
