writing-plans (编写计划)
发布于：2026年03月27日 更新于：2026年03月26日 22:16

这是一份为你翻译好的中文版本。保留了原文档的 Markdown 格式和代码块，对于专有技术名词（如 DRY, YAGNI, TDD, harness 等），我保留了英文或提供了对应的中文术语，以确保在开发环境中的准确性。

名称： writing-plans (编写计划)

描述： 在开始编写代码之前，当你拥有一个多步骤任务的规范说明（spec）或需求时使用。

编写计划

概述

编写全面的实施计划，假设执行工程师对我们的代码库毫无背景了解，且“品味存疑”。记录他们需要知道的一切信息：每个任务需要修改哪些文件、需要编写的代码、测试，他们可能需要查看哪些文档，以及如何进行测试。将整个计划拆解为极小粒度（bite-sized）的任务提供给他们。遵循 DRY（不重复造轮子）、YAGNI（不需要的不做）、TDD（测试驱动开发）原则。频繁提交代码。

假设他们是熟练的开发者，但对我们的工具集或问题领域几乎一无所知。假设他们不太懂得什么是好的测试设计。

在开始时声明： “我正在使用 writing-plans (编写计划) 技能来创建实施计划。”

上下文： 这应该在一个专用的工作树（worktree，由 brainstorming 头脑风暴技能创建）中运行。

将计划保存至： docs/superpowers/plans/YYYY-MM-DD-<功能名称>.md

（如果用户对计划存放位置有自定义偏好，则覆盖此默认设置）

范围检查 (Scope Check)

如果规范涵盖了多个独立的子系统，那么它应该在头脑风暴阶段就被拆分成多个子项目规范。如果没有，请建议将其拆分为独立的计划——每个子系统对应一个计划。每个计划都应该能独立产出可运行、可测试的软件。

文件结构 (File Structure)

在定义任务之前，先规划好将要创建或修改哪些文件，以及每个文件的职责是什么。这是敲定拆解决策的关键环节。

设计具有清晰边界和明确接口的单元。每个文件都应有单一明确的职责。

你能最好地理解并推理那些可以一次性容纳在上下文中的代码，且当文件功能专注时，你的修改也会更可靠。倾向于使用较小、职责集中的文件，而不是承担过多任务的大文件。

一起变动的文件应该放在一起。按职责进行拆分，而不是按技术层级拆分。

在现有的代码库中，请遵循已建立的模式。如果代码库使用的是大文件，不要单方面去重构它——但是，如果你正在修改的文件已经变得臃肿不堪，在计划中包含拆分该文件的步骤是合理的。

这种文件结构决定了任务的拆解方式。每个任务都应产生自洽的变更，且这些变更独立来看是有意义的。

极小粒度任务 (Bite-Sized Task Granularity)

每一步都是一个具体的动作（大约 2-5 分钟）：

  • “编写预期会失败的测试” - 步骤
  • “运行它以确保其失败” - 步骤
  • “实现最小代码以使测试通过” - 步骤
  • “运行测试并确保其通过” - 步骤
  • “提交 (Commit)” - 步骤

计划文档头部 (Plan Document Header)

每个计划必须以此头部开始：

Markdown

# [功能名称] 实施计划

> **对于智能代理（Agentic workers）：** 必填项：使用 `superpowers:subagent-driven-development`（如果子代理可用）或 `superpowers:executing-plans` 来实施此计划。步骤需使用复选框（`- [ ]`）语法进行跟踪。

**目标：** [用一句话描述这构建了什么]

**架构：** [用 2-3 句话说明实现方法]

**技术栈：** [关键技术/库]

---

任务结构 (Task Structure)

Markdown

### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写预期会失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected

  • [ ] 步骤 2：运行测试以验证其失败

    运行：pytest tests/path/test.py::test_name -v

    预期结果：失败（FAIL），提示 "function not defined"

  • [ ] 步骤 3：编写最小实现代码

Python

def function(input):
    return expected

  • [ ] 步骤 4：运行测试以验证其通过

    运行：pytest tests/path/test.py::test_name -v

    预期结果：通过（PASS）

  • [ ] 步骤 5：提交代码 (Commit)

Bash

git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
**注意事项 (Remember)**
* 始终使用精确的文件路径。
* 计划中需包含完整的代码（不要只写“添加验证”这种模糊的话）。
* 提供精确的执行命令和预期输出。
* 使用 `@` 语法引用相关的技能。
* 遵循 DRY、YAGNI、TDD 原则，频繁提交代码。

**计划审查循环 (Plan Review Loop)**
在编写完完整的计划后：
* 分发一个名为 `plan-document-reviewer` 的子代理（参见 `plan-document-reviewer-prompt.md`），并为其提供精心准备的审查上下文——**绝不能提供你的会话历史记录**。这能让审查者专注于计划本身，而不是你的思考过程。
* 提供信息：计划文档的路径、规范说明（spec）文档的路径。
* 如果 ❌ 发现问题：修复这些问题，并重新分发审查者来审查整个计划。
* 如果 ✅ 获得批准：进入执行交接阶段。

**审查循环指南：**
* 由编写该计划的同一个代理来修复问题（以保留上下文）。
* 如果循环超过 3 次迭代，请向人类用户求助以获取指导。
* 审查者的意见是建议性质的——如果你认为反馈不正确，请解释你的分歧原因。

**执行交接 (Execution Handoff)**
保存计划后：
* 向用户说：“计划已完成并保存至 `docs/superpowers/plans/<文件名>.md`。准备好执行了吗？”

执行路径取决于工具框架（harness）的能力：
* **如果框架支持子代理（如 Claude Code 等）：**
    * **必须：** 使用 `superpowers:subagent-driven-development`。
    * **不要**提供选择——子代理驱动是标准做法。
    * 每个任务使用一个全新的子代理 + 两阶段审查。
* **如果框架不支持子代理：**
    * 在当前会话中使用 `superpowers:executing-plans` 来执行计划。
    * 批量执行并设置检查点（checkpoints）以供审查。
